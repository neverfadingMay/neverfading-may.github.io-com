软件测试-3

发表于 2016-02-18   |     |   次阅读

本篇为软件测试系列-3

测试工程师（SDET, Software Development Engineer in Test)，主要职责包括

- 制定测试计划
- 设计测试用例
- 分析缺陷的根本原因
- 参与程序代码的审查
- 产品设计的审查
- 开发测试自动化程序

来回顾一下 传统软件工程模型
瀑布模型(又称“系统发展生命周期(System Development Life Cycle,SDLC)”)

最早强调系统开发应有完整之周期 ，且必须完整地经历周期之每一开发阶段，并系统化地考量分析与设计的技术、时间与资源投入等。

螺旋模型

最大特点在于引入了其他模型不具备的风险分析，使软件在无法排除重大风险时有机会停止。
包含了4个迭代过程

- 确定目标
- 风险评估
- 工程实现
- 下一迭代规划

敏捷开发

对比迭代式开发
周期更短，并更加强调队伍中的高度协作
对比瀑布式开发
强调的是尽早将尽量小的功能交付使用。

修哈特循环(PDCA循环)

  • Plan 事先计划、分析、确定过程和预测结果
  • Do 执行计划和过程
  • Check/Study 分析结果（后来改名为“学习”）
  • Act 审查所有步骤，并采取行动改善这一进程

好的第一部分终于结束了，总结第一部分就是微软公司在软件测试方面做出的努力和安排。进入到第二部分。

第二部分和第三部分也是大致看一下，就是对软件开发和测试的思考。来看第四部分。

第四部分：
使用测试模式。测试模式分享的常用的形式是模板。
模板包括以下属性：

  • 名称。 提供一个在对话中可以被谈及的容易记住的名字
  • 问题。 提供一句话来描述这个模式所能解决的问题
  • 分析。 描述问题的领域（或者提供一小段对问题领域的描述),并回答这个技术如何比简单的随意选择式好。
  • 设计。 解释这个模型如何被执行（模式如何从设计转变为测试用例）
  • 预言。 解释期望的结果（也可以包括在设计部分中）
  • 用例。 解释这个模式如何发现程序缺陷的例子。
  • 缺陷和局限。 解释在什么环境和情况下，应避免使用这个模式。
  • 相关的模式。 列出任何相关的模式（如果有的话）
